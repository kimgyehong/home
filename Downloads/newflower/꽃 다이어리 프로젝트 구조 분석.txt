🔗 의존성 흐름 (Dependency Flow)
상향식 의존성 (Bottom-Up)
platform:* → core:common → core:domain → core:data → feature:diary → ui:* → app:*
세부 의존성 관계
core:common (최하위 계층)

의존성: 없음 (순수 Kotlin만)
역할: 플랫폼 추상화, 상수, 유틸리티
platform:android/ios

의존성: core:common
역할: expect/actual 구현체 제공
core:domain

의존성: core:common
역할: 비즈니스 로직, 엔티티, 유스케이스
core:data

의존성: core:common + core:domain
역할: 데이터 저장소, SQLDelight 구현
feature:diary

의존성: core:common + core:domain + core:data
역할: 비즈니스 로직 조합, ViewModel
ui:android/ios

의존성: feature:diary + core:*
역할: 플랫폼별 UI 구현
app:android/ios

의존성: ui:* + platform:* + 모든 모듈
역할: 앱 런처, DI 설정

🎯 핵심 설계 원칙
1. 플랫폼 중립성 (Platform Neutrality)
core:*, feature:* 모듈은 100% 순수 Kotlin
Android/iOS 의존성 완전 배제
expect/actual 패턴으로 플랫폼 추상화
2. 단방향 의존성 (Unidirectional Dependency)
상위 계층만 하위 계층에 의존
순환 의존성 완전 차단
Clean Architecture 원칙 준수
3. 관심사 분리 (Separation of Concerns)
각 모듈이 명확한 단일 책임
비즈니스 로직과 UI 로직 완전 분리
데이터 계층과 도메인 계층 분리
4. 의존성 역전 (Dependency Inversion)
인터페이스에 의존, 구현체에 의존하지 않음
expect/actual 패턴으로 플랫폼 추상화
Repository 패턴으로 데이터 소스 추상화
