  ModuleScaleManager.kt - 모듈 크기 자동 관리

  // buildSrc/src/main/kotlin/ModuleScaleManager.kt

  object ModuleScaleManager {
      private const val MAX_FILES_PER_MODULE = 75
      private const val WARNING_FILES_PER_MODULE = 60

      fun checkModuleSize(project: Project) {
          val sourceFiles = project.fileTree("src").matching {
              include("**/*.kt")
          }.files

          when {
              sourceFiles.size > MAX_FILES_PER_MODULE -> {
                  throw GradleException("""
                      ❌ Module ${project.name} has ${sourceFiles.size} files (max: $MAX_FILES_PER_MODULE)

                      🔧 Consider splitting into submodules:
                      - ${project.name}-domain
                      - ${project.name}-data
                      - ${project.name}-ui

                      📖 See docs/module-split-guide.md for details
                  """.trimIndent())
              }

              sourceFiles.size > WARNING_FILES_PER_MODULE -> {
                  project.logger.warn("""
                      ⚠️  Module ${project.name} approaching size limit: ${sourceFiles.size}/$MAX_FILES_PER_MODULE
  files
                      Consider preparing for module split
                  """.trimIndent())
              }
          }
      }
  }

  // build.gradle.kts에서 자동 체크
  tasks.register("checkModuleSize") {
      doLast {
          ModuleScaleManager.checkModuleSize(project)
      }
  }

  tasks.named("compileKotlin") {
      dependsOn("checkModuleSize")
  }

  ## 🔥 해결책 2: 하이브리드 복잡성 자동 해결

  ### ArchitectureRules.kt - 일관성 자동 강제
  ```kotlin
  // buildSrc/src/main/kotlin/ArchitectureRules.kt

  object ArchitectureRules {

      // 🔥 타입 시스템 규칙 자동 검증
      fun validateTypeConsistency(project: Project) {
          val sourceFiles = project.fileTree("src").matching {
              include("**/*.kt")
          }.files

          sourceFiles.forEach { file ->
              val content = file.readText()
              val packageName = extractPackage(content)

              when {
                  packageName.contains(".model.") -> {
                      // 도메인 모델은 반드시 data class + validation
                      if (!content.contains("data class") || !content.contains("require(")) {
                          throw GradleException("""
                              ❌ Domain model in ${file.name} must use:
                              - data class (not value class)
                              - init { require(...) } validation

                              📖 See architecture-decisions.md #domain-types
                          """.trimIndent())
                      }
                  }

                  packageName.contains(".types.") -> {
                      // ID 타입은 반드시 sealed interface + data class
                      if (content.contains("@JvmInline") || content.contains("value class")) {
                          throw GradleException("""
                              ❌ ID types in ${file.name} must use:
                              - sealed interface EntityId
                              - data class FooId(...) : EntityId

                              📖 See architecture-decisions.md #id-types
                          """.trimIndent())
                      }
                  }

                  packageName.contains(".store.") -> {
                      // Store는 반드시 완전 불변
                      if (content.contains("mutableListOf") || content.contains("mutableMapOf")) {
                          throw GradleException("""
                              ❌ Store in ${file.name} must use:
                              - ImmutableList, ImmutableMap
                              - No mutable collections

                              📖 See architecture-decisions.md #store-immutability
                          """.trimIndent())
                      }
                  }
              }
          }
      }

      private fun extractPackage(content: String): String {
          return content.lines()
              .find { it.startsWith("package ") }
              ?.substringAfter("package ")
              ?.substringBefore(";")
              ?: ""
      }
  }

  // build.gradle.kts에서 자동 검증
  tasks.register("validateArchitecture") {
      doLast {
          ArchitectureRules.validateTypeConsistency(project)
      }
  }

  tasks.named("compileKotlin") {
      dependsOn("validateArchitecture")
  }

  architecture-decisions.md - 모든 결정 문서화

  # 아키텍처 결정 기록

  ## #domain-types: 도메인 타입 시스템

  ### 결정: data class + validation 사용
  ```kotlin
  // ✅ 올바른 방법
  data class DiaryId(val value: String) : EntityId {
      init {
          require(value.startsWith("diary-")) { "Invalid DiaryId" }
      }
  }

  // ❌ 잘못된 방법
  @JvmInline value class DiaryId(val value: String)
  typealias DiaryId = String

  이유:

  1. KMP 완전 호환: 모든 플랫폼에서 동일한 동작
  2. Serialization 안전: kotlinx-serialization 완벽 지원
  3. 타입 안전성: 컴파일 타임 + 런타임 모두 보장
  4. 코드 중복 방지: expect/actual 불필요

  언제 사용:

  - 모든 도메인 엔티티 ID (DiaryId, FlowerId, UserId)
  - 핵심 비즈니스 로직에서 사용되는 타입

  #performance-types: 성능 최적화 타입

  결정: typealias 사용

  // ✅ 성능 중요 영역
  typealias SearchQuery = String
  typealias FilterKey = String
  typealias SortField = String

  이유:

  1. 성능 우선: 메모리 오버헤드 Zero
  2. 빈번한 생성: 검색/필터링에서 자주 생성됨
  3. 타입 혼동 허용: 성능을 위해 일부 안전성 포기

  언제 사용:

  - 검색, 필터링, 정렬 관련 타입
  - UI 상태 관련 임시 타입

  #store-immutability: Store 불변성

  결정: 완전 불변 + 제어된 캐시

  // ✅ 핵심 상태는 완전 불변
  data class DiaryState(
      val diaries: ImmutableList<Diary> = persistentListOf()
  )

  // ✅ 성능 캐시는 제어된 가변성
  private val searchCache = mutableMapOf<String, List<Diary>>()
  private val cacheMutex = Mutex()

  이유:

  1. 안전성: 핵심 상태는 절대 변경 불가
  2. 성능: 캐시는 Mutex로 보호된 가변성 허용
  3. 격리: 캐시 변경이 핵심 상태에 영향 없음

  언제 적용:

  - 모든 Store의 핵심 상태: 완전 불변
  - 성능 최적화 캐시: 제어된 가변성 허용

  ## 🤔 자주 묻는 질문

  ### Q: 왜 DiaryId는 data class인데 SearchQuery는 typealias인가요?
  A: **성능 vs 안전성** 트레이드오프입니다.
  - DiaryId: 도메인 핵심이므로 안전성 우선 → data class
  - SearchQuery: 성능 중요하므로 속도 우선 → typealias

  ### Q: 언제 모듈을 분할해야 하나요?
  A: **75개 파일 초과 시** 자동으로 빌드가 실패합니다.
  `docs/module-split-guide.md`를 참고하여 분할하세요.

  ### Q: Store에서 mutableMap을 쓰면 안 되나요?
  A: **핵심 상태**에서는 절대 안 됩니다. **캐시**에서만 Mutex와 함께 허용됩니다.
